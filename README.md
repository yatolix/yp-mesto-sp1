# Уровень 1 (Проектирование)

**ВАЖНО**: Уровень 1 (проектирование) описан по шагам процесса https://code.s3.yandex.net/software-architect/Как%20разбить%20монолитный%20фронтенд%20на%20микрофронтенды.pdf.  

## 1. Предварительный анализ

Предварительный анализ исходного монолитного приложения показал, что приложение:

- достаточное простое с точки зрения размера
- у приложения несколько разных бизнес-функций, одна из которых является главной (отображение мест с фотографиями). И несколько второстепенных: регистрация и логин, управление профилем пользователя, проставление лайков

Не смотря на простоту, имеет смысл разделить приложение на микрофронтенды. Это позволит сразу же развивать отдельные функции приложения независимо друг от друга. В частности основная функция приложения может получить мощное развитие. А второстепенные (логин, управление профилем) скорее всего будут меняться относительно редко. Разделение на микрофронтенды позволит сделать релизные циклы бизнес-функций приложения независимыми и более гибко масштабировать каждую часть приложения.

## 2. Определение бизнес-функций

В качестве стратегии разделения на микрофронтенды будет применена вертикальная нарезка, потому что она позволит независимо развивать отдельные бизнес-функции. Другие стратегии в свою очередь не дают ощутимых приемуществ.

Стратегия автономности команд тут не применима, потому что приложение небольшое и скорее всего развивать его будет пока что одна команда.

Стратегия изоляции также несёт мало смысла, потому что в приложении достаточно простой интерфейс состоящий из базовых компонентов, которые можно переиспользовать в разных микрофронтендах - для изоляции их пришлось бы дублировать, что пока не целесообразно.

Итак, мы применяем вертикальную нарезку и выделяем 3 бизнес-функции:

- галерея мест с возможностями добавлять/удалять их, ставить лайки
- аутентификация и вторизация, включающая в себя логин и регистрацию
- управление профилем пользователя, в то числе смена аватара

## 3. Выбор метода реализации

### 3.1 Метод интеграции - Run time 

Ожидается, что бизнес-функции будут развиваться с разной интенсивностью и будут обладать отличными друг от друга релизными циклами. Для реализации этих ожиданий и требований был выбран метод интеграции **Run time**.

**Build time** нам тут мало подойдёт, потому что у нас пока не стоит задачи оптимизировать производительность и сохранить тесное взаимодействие между функциями. А вот динамическое обновление модулей и их гибкое масштабирование будут весьма кстати, что как раз является плюсом **Run time**.

### 3.2 Метод компоновки - клиентская 

Даже в текущей версии приложения уже существует набор виджетов которые могут взаимодействовать между собой. Также есть виджеты, которые при первоначальной загрузке скрыты от пользователя (редактирование профиля к примеру), что даёт возможность грузить их по мере неоходимости. Как следствие, нам в качестве метда компоновки подходит **Клиентская**.

## 4. Выбор инструмента
====

В качестве инструмента был выбран **Webpack Module Federation**. Он позволяет "шарить" библиотеки между микрофронтендами. Например, общие стили, иконки и универсальные popup-ы. **Single SPA** тоже подошёл бы, но его приемущества не сыграют в условиях нашего приложения - у нас пока одна команда и нет необходимости использовать несколько технолгий одновременно. Single SPA больше бы подошёл для реализации единого рабочего места сотрудников, которое представляет из себя портал с разными модулями, которые разрабатываются назвисимыми командами.

## 5. Выбор стратегии управления состоянием 

В качестве стратегии управления состоянием будет использоваться "глобальное состояние". Приложение написано на React, который относительно просто позволяет управлять глобальным состоянием. Только в каких-то частных случаях возможно использовать подход Pub-Sub.

# Уровень 2 (Планирование изменений)

## Описание Mesto на базе микрофронтендов

Обновлённая версия приложения Mesto состоит из следующих микрофронтендов и компонентов:

- **host-mf**: корневое приложение, которое содержит общие компоненты (css, popup's  и т.д). Также в этом приложении содержится главный шаблон, в котором задаётся структура виджетов из других микрофронтендов.
   - App (общая структура расположения компонентов из разных микрофронтендов)
   - Main (отображение контента для залогиненного пользователя)
   - Footer (общий футер)
   - Header (общая "шапка")
   - PopupWithForm (Универсальный попап который использется в разных микрофронтендах. Микрофронтенды могут импортировать его из корневого модуля)
   - ProtectedRoute (общий компонент чтобы обеспечить доступ к защищённой части сайта)
- **auth-mf**: микрофронтенд отвечающий за регистрацию, логин (в т.ч. работа с токеном).
   - InfoTooltip (Уведомление об успешной регистрации. Используется только в Регистрации. Содержит текстовки специфичные для регистрации, поэтому без изменениц его невозможно переиспользовать)
   - Login (Форма логина)
   - Register (Форма регистрации)
- **places-cards-mf**: микрофронтенд реализующий основную бизнес-функцию приложения - галерею мест и управление ею. Здесь содержатся карточки мест, лайки, а так же функции управления местами (просмотр, добавление, удаление).
   - AddPlacePopup (форма для добавления новых мест)
   - Card (компонент для отображения карточки с изображением места, описанием и возможностью удалить или лайкнуть место)
   - ImagePopup (попап для просмотра полного изображения с местом)
- **profile-mf**: микрофронтенд для работы с профилем посетителя. Отвечает за редактирование профиля, в том числе обновление аватара.
   - EditAvatarPopup (форма редактирования автара)
   - EditProfilePopup (форма редактирования профиля пользователя)

Данное разделение имеет под собой ряд обоснований:

- **auth-mf** - это в первую очередь про безопасность: аутентификация, права доступа, токены и т.д. Мехаинзмы, заложенные там, меняются как правило нечасто. А если и меняются, то обусловлены сменой стратегии безопасности или обновлением стандартов. Например, в будущем может появиться потребность интеграции с Open ID провайдером или добавление двухфакторной аутентификации. Все эти дополнения можно будет реализовывать отдельно от бизнесовых функций приложения.
- **profile-mf** и **auth-mf** целесообразнее развивать по отделььности . Они хоть и связаны логически (и там и там работаем с данными пользователя), но **profile-mf** - это не про безопасность, а про бизнесовую составляющую. Поводы для изменений **profile-mf** будут координально отличаться от изменений в **auth-mf**. Например, в **profile-mf** можно добавить туристические предпочтения пользователей. Эти изменения никак не должны повлиять на аутентификацию.
- можно было бы вынести работы с аватаром в отдельный микрофронтенд, но на первоначальном этапе этого точно делать не следует. По бизнес смыслу, редактирование аватара тесно связано в профилем пользователя.
- в **places-cards-mf** помимо самих карточек с местами есть "лайки". В текущей реализации лайки тесно связаны с карточками мест. Если мы выделим "лайки" в отдельный микрофронтенд, то это может привести к усложнению логики, потому что нам надо будет синхронизировать состояние между карточками и лайками. Но в будущем допустимо пересмотреть данное решение, так как функционал лайков может развиваться, туда могут добавиться комментарии, смайлики, рейтинги и т.п. В совокупности всё перечисленное - это **реакции** пользователя. Если возникнет бизнес-потребность, то реакции могут стать вполне себе самостоятельным микрофронтендом.
- **host-mf** нам нужен не только как хранитель общего визуального шаблона виджетов, но и как источник общих компонентов и их стилей.

Отдельно стоит остановиться на моментах, которые необходимо учесть при реализации Уровня 3 задания:

- разобраться с подгрузкой данных по карточкам и пользователям в общий контекст приложения при старте. Кажется, что должна быть лейзи загрузка в соотв. микрофронтендах.
- все функции обработчики кликов должны быть аккуратно распределены по микрофронтендам
- данные о пользователи нужны как auth-mf так и profile-mf - нужно придумать как корректно их шарить. Будет ли какой-то конкретный компонент отвечать за их загрузку, а второй - переиспользовать. Или загрузка должна происходить в корневом компонента, что неочень корректно.
- прописать exposes и remotes для общих компонентов и стилей из host-mf
- чуть глубже проанализировать в host-mf HTML структуру оборачивающих элементов для виджетов других микрофронтендов. Есть ощущение, что часть обёрток можно вынести в микрофронтенды. Например список places__list кажется не должен быть в host-mf.

## Запуск приложения

!!! Запуск будет возможен после выполнения Уровня 3 задания !!!

### auth-mf
1. Из корневой папки проекта открываем терминал и выполняем `cd frontend/auth-mf`
2. выполняем `npm install`
3. выполняем `npm start`
4. открываем в браузере `http://localhost:8081`.

### profile-mf
1. Из корневой папки проекта открываем терминал и выполняем `cd frontend/profile-mf`
2. выполняем `npm install`
3. выполняем `npm start`
4. открываем в браузере `http://localhost:8082`.

### places-cards-mf
1. Из корневой папки проекта открываем терминал и выполняем `cd frontend/places-cards-mf`
2. выполняем `npm install`
3. выполняем `npm start`
4. открываем в браузере `http://localhost:8083`.

### host-mf
1. Для корректной работы желательно сначала запустить все остальные микрофронтенды, а потом запускать корневой `host-mf`
2. Из корневой папки проекта открываем терминал и выполняем `cd frontend/host-mf`
2. выполняем `npm install`
3. выполняем `npm start`
4. открываем в браузере `http://localhost:8080`.
